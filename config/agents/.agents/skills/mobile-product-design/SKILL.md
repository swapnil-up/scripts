# Mobile Product Design

Design mobile apps that feel like real products.

The goal is not novelty.

The goal is clarity, trust, and ease of use.

A good screen should be understood in a few seconds.

A good app should feel obvious, calm, and efficient.

## Core Principle

Every screen must answer four questions:

1. What is this screen for?
2. What is the primary action?
3. What is the minimum UI needed?
4. What happens when data is missing, loading, or broken?

If the answer is unclear, simplify the screen.

## Design Priorities

Use this order:

1. Clarity
2. Hierarchy
3. Usability
4. Consistency
5. Visual polish
6. Novelty

Never trade the first four for the last two.

## Default Approach

Prefer familiar mobile patterns.

Remove complexity before adding visual treatment.

Design for repeated use, not first-time surprise.

If something looks clever but slows the user down, it is wrong.

## Screen Rules

Each screen should have one main job.

Each screen should have one obvious primary action.

Each screen should have one visual focus.

If a screen needs to do too much, split it.

If a screen has multiple equal actions, the hierarchy is weak.

## Layout Rules

Use a simple flow:

context, content, action.

Design for vertical scanning.

Use spacing to create structure.

Do not make every element compete for attention.

Most mobile screens should feel lighter than they first seem possible.

## Component Rules

Use lists when scanning matters.

Use cards only when grouping adds meaning.

Use buttons intentionally.

Use icons to support labels, not replace them.

Use forms that feel lightweight.

Avoid custom controls unless they solve a real problem better than the standard pattern.

## Form Rules

Forms should minimize effort.

Prefer defaults, pickers, autofill, and progressive disclosure.

Do not ask for information unless it is needed.

Keep validation calm and specific.

Long forms should be broken into steps.

## State Rules

Every important screen must handle these well:

- Empty state
- Loading state
- Error state
- Success state

Empty states should explain what belongs there and what to do next.

Loading states should never feel like dead air.

Error states should explain the problem and the next step.

Success states should confirm completion clearly.

## Navigation Rules

Use standard mobile navigation patterns.

Prefer shallow hierarchy.

Prefer predictable back behavior.

Prefer bottom navigation for primary areas when appropriate.

Avoid creative navigation schemes.

Users should not have to learn the app before using it.

## Settings and Management Screens

Group settings logically.

Use clear sections.

Split long pages into smaller pages.

Do not create one giant settings dump.

If a settings screen feels endless, the structure is wrong.

## Visual Style

Typography should create hierarchy and remain highly readable.

Color should communicate brand, state, and priority.

Spacing should do more work than shadows.

Decoration should never compete with usability.

Make the interface look polished, but do not make it decorative for its own sake.

## Mobile Usability

Design for thumbs.

Make touch targets comfortable.

Avoid precision-heavy interactions.

Keep important actions reachable.

One-handed use should be possible whenever reasonable.

## Accessibility

Assume:

- large text
- reduced motion
- screen readers
- limited dexterity
- small screens
- low attention

Accessibility is part of good design.

## Consistency Rules

Reuse patterns aggressively.

If the user learns a pattern once, use it everywhere.

Consistency builds trust.

Inconsistency feels amateur.

## Quality Benchmarks

Use these products as references for clarity and product maturity:

- Linear
- Notion
- Monzo
- Revolut
- Todoist
- GitHub Mobile
- Spotify
- Google Photos
- Apple Health
- Airbnb

Do not copy their visuals.

Copy their clarity, hierarchy, and restraint.

## Design Questions

Before finalizing a screen, ask:

1. What is the primary action?
2. What can be removed?
3. Is the hierarchy obvious?
4. Is the screen doing too much?
5. What happens with no data?
6. What happens while loading?
7. What happens on error?
8. Can this be used one-handed?
9. Does it feel consistent with the rest of the app?
10. Does it feel like a shipped product?

If any answer is weak, refine the design.

## Red Flags

Reject the design if it has:

- multiple competing primary actions
- crowded layouts
- excessive cards
- weak hierarchy
- hidden functionality
- tiny touch targets
- unclear states
- long confusing forms
- deep nested navigation
- too much decoration
- dashboard syndrome

If the user needs an explanation, the design is not ready.

## Output Standard

The final result should feel like a real app shipped by a serious team.

It should be clear, modern, trustworthy, polished, and easy to use.

Make the app feel competent before making it feel distinctive.
